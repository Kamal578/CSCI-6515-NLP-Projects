\begin{abstract}
This project implements two neural NLP systems and an interactive interface. Task 1 fine-tunes a BERT-based classifier for Azerbaijani review sentiment using the Hugging Face dataset \texttt{hajili/azerbaijani\_review\_sentiment\_classification}. Task 2 implements extractive reading comprehension with a PyTorch BiDAF architecture and compares two embedding settings: trainable GloVe vectors and frozen BERT embeddings. The final deliverable includes a Streamlit UI that can reload the saved local checkpoints and run live inference for both tasks on macOS Apple Silicon with MPS acceleration when available.
\end{abstract}

\section{Motivation}
Azerbaijani is underrepresented in mainstream NLP tooling, and its agglutinative morphology makes simple word-level baselines fragile. Sentiment-bearing words often appear with long suffix chains, informal spelling, and app-review noise. This motivates a subword-based model for Task 1. For Task 2, the project studies whether contextual BERT embeddings improve a classical BiDAF span extractor compared with traditional word vectors. Together, the tasks cover sentence classification, span prediction, and deployment-oriented interface design.

\section{Task 1: Azerbaijani Sentiment Analysis}
\subsection{Dataset}
Task 1 uses the public Hugging Face dataset \texttt{hajili/azerbaijani\_review\_sentiment\_classification}. The dataset contains Azerbaijani review text in the \texttt{content} column, numeric review scores in the \texttt{score} column, and \texttt{upvotes}. The current dataset version has a \texttt{train} split with 127{,}537 rows and a \texttt{test} split with 31{,}885 rows. Our implementation can train in three label modes:
\begin{itemize}
    \item \texttt{score5}: original 1--5 star labels,
    \item \texttt{sentiment3}: negative / neutral / positive,
    \item \texttt{binary}: negative vs.\ positive after dropping score 3.
\end{itemize}
The default configuration keeps the original five-class formulation.

\subsection{Model Choice}
The default Task 1 model is \texttt{google-bert/bert-base-multilingual-cased}. This choice is pragmatic: it is stable, standard, and directly supported by \texttt{transformers}. The code also supports Azerbaijani-specific checkpoints such as \texttt{HPLT/hplt\_bert\_base\_az} when the user enables \texttt{trust\_remote\_code}. The implementation therefore balances accessibility and Azerbaijani specialization rather than hard-coding a single gated model.

\subsection{Preprocessing and Morphology}
Task 1 normalizes Unicode to NFC, cleans whitespace and review noise, standardizes punctuation variants, and preserves Azerbaijani diacritics. We intentionally do not stem or strip suffixes, because such heuristics can remove sentiment-bearing morphology. Instead, we rely on BERT subword tokenization as the main agglutinative adaptation strategy. The training script also writes morphology diagnostics to \texttt{summary.json}, including frequent suffix endings and tokenizer fragmentation statistics such as average wordpieces per word.

\subsection{Inputs and Outputs}
The model input is a single Azerbaijani review string. After tokenization, the tensor inputs are the standard BERT classification tensors: \texttt{input\_ids}, \texttt{attention\_mask}, and when the tokenizer provides it, \texttt{token\_type\_ids}. The output is a vector of class logits. Prediction is obtained with an argmax over the logits. The exact label names, case-sensitivity status, and configured maximum input length are recorded in \texttt{outputs/project4/task1\_sentiment/summary.json}.

\subsection{Training Setup}
The implementation is written in PyTorch and \texttt{transformers}. It supports Apple Silicon MPS automatically and falls back to CPU if MPS is unavailable. To avoid memory issues on a 16~GB MacBook, the code uses dynamic padding, configurable maximum sequence length, gradient accumulation, and early stopping on validation macro-F1. The training entry point is:
\begin{verbatim}
bash scripts/run_project4_task1.sh
\end{verbatim}

\subsection{Evaluation}
Task 1 reports accuracy, macro-F1, weighted F1, a confusion matrix, and a classwise classification report. The exact numbers are generated automatically into the output directory after training. For submission, the final reported values should be copied from:
\begin{itemize}
    \item \texttt{outputs/project4/task1\_sentiment/summary.json}
    \item \texttt{outputs/project4/task1\_sentiment/history.csv}
    \item \texttt{outputs/project4/task1\_sentiment/test\_predictions.json}
\end{itemize}

\section{Task 2: Reading Comprehension with BiDAF and BERT}
\subsection{Architecture}
Task 2 implements a shared BiDAF span-prediction backbone in PyTorch. The system contains:
\begin{itemize}
    \item a two-layer highway network,
    \item contextual BiLSTM encoding,
    \item bidirectional attention flow between context and question,
    \item modeling layers,
    \item separate start and end span prediction heads.
\end{itemize}
Two embedding variants are supported:
\begin{itemize}
    \item \textbf{BiDAF + GloVe}: trainable word embeddings loaded from pretrained vectors,
    \item \textbf{BiDAF + frozen BERT}: contextual word embeddings pooled from a BERT encoder and projected into the BiDAF stack.
\end{itemize}

\subsection{Dataset and Preprocessing}
The default dataset is SQuAD 1.1 loaded through the \texttt{datasets} library. The code can also read local SQuAD-format JSON files. Long contexts are split into overlapping windows. During training, only windows that contain the gold answer span are kept. During evaluation and UI inference, all valid windows are scored and the best span is selected globally.

\subsection{Training and Metrics}
The Task 2 runner is:
\begin{verbatim}
bash scripts/run_project4_task2.sh
\end{verbatim}
The code supports \texttt{glove}, \texttt{bert}, or \texttt{compare} mode, MPS or CPU execution, gradient accumulation, and checkpoint saving. Evaluation uses SQuAD-style Exact Match (EM) and token-level F1. The comparison artifacts are written to:
\begin{itemize}
    \item \texttt{outputs/project4/task2\_reading\_comprehension/comparison.csv}
    \item \texttt{outputs/project4/task2\_reading\_comprehension/report\_notes.md}
    \item per-variant \texttt{summary.json}, \texttt{history.csv}, and \texttt{predictions.json}
\end{itemize}
For the final submitted report, the best EM and F1 values should be copied from \texttt{comparison.csv}.

\subsection{Expected Comparison}
The central experimental question is whether contextual BERT embeddings help BiDAF more than traditional word vectors. The implemented comparison is designed so the architecture above the embedding layer stays fixed, making the GloVe vs.\ frozen-BERT difference interpretable. In most realistic runs we expect the frozen-BERT variant to outperform the GloVe baseline on validation F1, although the exact size of the gain depends on the subset size and training budget.

\section{Task 3: Interactive UI}
The Streamlit interface is launched with:
\begin{verbatim}
bash scripts/run_project4_ui.sh
\end{verbatim}
The UI supports:
\begin{itemize}
    \item live Azerbaijani sentiment prediction from free-form text,
    \item live QA answer extraction from a user-supplied context and question,
    \item metric and artifact inspection,
    \item report preview.
\end{itemize}
The sentiment tab reloads the locally fine-tuned Task 1 checkpoint and displays class probabilities. The QA tab reloads the saved BiDAF checkpoints and runs answer extraction through the same preprocessing and span-selection logic used during evaluation.

\section{MacBook Pro M2 Pro Optimization}
The project targets a MacBook Pro M2 Pro with 16~GB RAM. The implementation uses the following practical constraints:
\begin{itemize}
    \item MPS is selected automatically when available,
    \item dynamic padding reduces wasted memory for short reviews,
    \item gradient accumulation allows smaller physical batch sizes,
    \item sequence lengths are capped aggressively during smoke runs,
    \item checkpoints and intermediate CSV/JSON artifacts are written incrementally.
\end{itemize}
The recommended workflow is to start with smoke runs for both tasks, confirm the pipelines end to end, and only then scale the dataset size or epoch count.

\section{Error Analysis Guidance}
For Task 1, typical failure cases are noisy spelling, sarcasm, mixed-polarity reviews, and score/text mismatch in the source data. For Task 2, common errors include selecting a nearby span instead of the exact gold answer, truncation failures in long contexts, and lexical overlap biases in the GloVe baseline. The saved prediction artifacts are intended to support manual error analysis during report writing and presentation preparation.
